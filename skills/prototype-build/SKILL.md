# 6단계 · 프로토타이핑 (prototype-build)

## 목적
솔루션 MVP를 **실제로 클릭되는 데모**(`prototype.html`)로 만든다. 발표 덱은 이 스킬이 아니라 **`pitch-deck-build`** 가 담당한다. 오피스아워 피드백으로 **여러 번 반복**한다(2박3일 동안 충분히 다듬는다).

## 시작 전
- `context.md`의 `주제`·`지역`·`데이터 검증 요약`·`아이디어`(사용자·시나리오·MVP)를 읽는다.
- 4단계에서 받은 실제 수치·장소명을 최대한 넣는다. 없으면 현실적인 예시값 + "예시" 표기.
- **디자인 품질은 `prototype-craft` 기준을 반드시 적용.** 설치돼 있으면 `theme-factory`·`brand-guidelines`도 얹는다.

## 먼저 묻기 (필수 — 퍼실리테이션)
바로 코드를 뽑지 말고 팀의 그림을 끌어낸다:
1. *"머릿속에 어떤 화면이 그려져요? 어떤 느낌·색이면 좋겠어요?"* (개방형) → `prototype-craft` 디자인 브리프(무드·색·기기)를 **팀이 정하게**. 이미 정했으면 생략.
2. **꼭 보여줄 핵심 기능 1개**를 팀이 짚게(아이디어의 MVP와 일치 확인).
답·합의는 `context.md`에 남긴다.

## 산출 규칙 (중요)
- **단일 파일 HTML**: HTML+CSS+JS 한 파일. 외부는 CDN만. **localStorage 등 브라우저 저장소 금지** — 상태는 JS 변수/메모리.
- 모바일에서도 보이게 반응형. 한국어 UI.
- 데이터는 파일 안에 JS 객체로 넣어 **오프라인에서도 데모**되게(발표장 네트워크 대비).

## prototype.html
- MVP 흐름(입력→처리→출력)을 **실제로 클릭되는** 화면으로. 최소 1개 핵심 기능이 동작.
- 예: 관심사 선택 → (내장 데이터로) 코스 생성 → 지도/카드 출력.
- 과욕 금지: 발표에서 보여줄 **1~2 기능만 완성도 있게.**

## 반복 (충분히 다듬기)
- 오피스아워·팀 피드백을 받으면 *"어디를 바꾸고 싶어요?"* 로 받아 **해당 부분만** 수정(전면 재작성 지양).
- 매 반복 후 `context.md` 로그에 "vN: 무엇을 고침" 기록. 시간이 남으면 실데이터·인터랙션·완성도를 계속 올린다.

## 심화 (시간 있으면 — 2박3일 깊이)
- **사용자 테스트**: 다른 팀·주변 사람에게 실제로 눌러보게 하고 *"어디서 헷갈렸어요?"* → 반영. (`field-research`와 연계)
- **실데이터 확장**: TourAPI에서 더 많은 장소·카테고리를 받아 데모를 풍부하게.
- **완성도 올리기**: 반응형·빈 상태·로딩·마이크로 인터랙션·접근성(대비·글자 크기).
- **2~3차 iteration**: 오피스아워 사이에 계속 다듬는다(한 번에 끝내지 않는다).

## context.md 갱신
- `산출물`에 `prototype.html` 경로.
- `단계: 6`, 로그 추가.

## 끝맺음 (여기까지가 4단계: 프로토타입)
- 요약: 파일 경로 + 핵심 기능 한 줄.
- **다음(피칭덱) 단계 스킬이 보이면** `$pitch-deck-build`로 이어가고, **아직이면** 열릴 때까지 재촉하지 말고 팀과 아래를 더 나눈다:
  - *"이 데모, 처음 보는 사람이 눌러도 30초 안에 '오' 할까요? 어디가 약해요?"*
  - *"다른 팀·주변 사람한테 한번 눌러보게 하고 반응을 볼까요?"* (`$field-research`)
  - *"발표 때 이 화면으로 어떤 이야기를 할지 미리 말로 해볼까요?"*
  - 시간 남으면 실데이터·완성도를 더 올린다(위 '심화').

## 톤·품질
- 완성도 > 기능 수. 하나라도 확실히 되는 데모.
